# 原型设计规格生成器

## 目的

将需求文档中的功能描述转化为逐页的原型设计规格文档。每页一个独立MD文件，
结构统一、可直接交付给研发和原型开发人员。

## 触发条件

- 用户说"设计页面功能规格""生成原型设计规格""按页面出规格""功能架构清单"
- 用户说"严格按需求文档设计""逐页面确认后再下一页"
- 场景：B端管理后台项目，功能来源为需求文档（招投标总表、PRD、业务流程文档等）

## 前置步骤

在开始逐页出规格前，必须先与用户对齐以下事项，不可跳过：

0. **记录需求来源**：确认并记录需求文档的路径或名称（如 `_source/xxx招投标总表.md`、`产品需求文档PRD.md`），该来源将写入每页规格文档的头部引用区域，确保规格可溯源到具体需求文档
1. **端侧确认**：梳理端侧划分，将需求功能逐项分配到端。B端项目通常为门户/门户个人中心/管理后台三端；C端项目可能含H5/小程序/APP。注意不同项目可能有不同端侧模型，必须与用户确认
2. **（所有端态）UI基线确认**：根据端侧确认结果，从设计库加载对应基线参数：
   - 2a. 查找设计库：读取 `01-通用规则/设计库/` 中对应端态的 design.md（web后台/web门户/手机端）
   - 2b. 确定主色系：设计库中为默认主色（web后台 #1E63B2、手机端 #3079ED）。若项目有自定义主色，提取项目色覆盖默认值
   - 2c. 读取共享约定：读取 `01-通用规则/设计库/设计约定.md`，确认跨项目共享约束（圆角规范、阴影层级、画板宽度、原型文件命名规则等）
   - 2d. （C端/手机端额外步骤）收集参考截图、确认页面类型、生成风格预览页——用户确认后进入逐页规格
3. **需求逐句拆解校验**：对需求原文逐句拆解功能点。每条功能描述必须能从需求文档找到对应语句——在规格中标注能力来源（如"来自需求第X条"）。无法直接对应的推断性功能必须明确标注"推断"并等用户确认后才能纳入规格，不得直接编造。注意区分"需求描述的功能"和"业务闭环缺口"——后者也需标注并经确认
4. **模块对齐**：一级模块名严格使用需求文档的模块名称
5. **业务流程**：画出完整业务流程图，明确每页在流程中的位置和上下游关系。此为必做步骤——未确认流程图不进逐页
6. **页面划分**：在流程图基础上逐模块确定页面数量和每页承载的功能编号，识别业务闭环缺口（需求文档未覆盖但业务必须的功能）
7. **已有能力复用**：确认平台已有能力（如流程引擎、系统消息、认证组件），避免重复设计已有功能
8. **产出架构清单**：输出功能架构与页面清单作为总纲，后续页面规格索引至此。架构清单必须含端/模块/页面/功能四级映射表

## 逐页生成流程

**每页流程（不可跳过）**：

1. 读取需求文档该页面对应的功能描述原文
2. 严格按功能描述输出规格，不自创功能、不自加字段
3. 输出后等待用户确认，确认通过后再进入下一页
4. 用户提出修改时，先改后确认，不跳过

## 文档模板规范

完整的模板和已生成的范本参考见 `references/template-structure.md`。
核心规范如下：

### 文件命名

`{页面编号}-{页面名称}-原型设计规格.md`，如 `M3-服务接口配置-原型设计规格.md`

### 章节结构（固定顺序）

```
# [编号] [页面名称] — 原型设计规格

> 端 / 模块 / 操作角色 / 承载功能编号
> 需求来源：[需求文档路径或名称]

## 零、功能概述
## 一、权限说明（含数据权限 + 功能权限）
## 二、页面结构
## 三至末尾、页面内容（按区域逐级展开）
## 末、交互规则
```

### 编号规范

- 大章节：一、二、三
- 子节：3.1、3.2
- 表格内首列固定为序号列
- 弹窗层级标注：L0(列表) / L1(侧滑 z-1000) / L2(弹窗 z-1050) / L3(确认 z-1100)

### 表格类型

| 表格用途 | 固定列 |
|---------|------|
| 功能权限 | 序号 / 功能权限 / 说明 |
| 列表表格 | 序号 / 列名 / 说明 |
| 操作行为说明 | 序号 / 操作 / 弹窗层级 / 行为 |
| 状态对照 | 序号 / 状态 / 颜色 / 说明 / 后续操作（或可用操作） |
| 字段配置 | 序号 / 字段 / 类型 / 必填 / 说明 |
| 交互规则 | 序号 / 规则 / 说明 |

### 功能概述必须包含

1. 核心业务——这页做什么、为什么做（2-3句）
2. 关键概念（如有）——数据模型关系、状态流转逻辑
3. 与上下游关系——输入从哪来、输出到哪些页面使用

### 操作与状态联动

- 操作按状态动态变化，用状态与操作对应表格表达
- 操作行为说明统一表格格式，含弹窗层级列
- 颜色规范：灰(草稿/禁用)、橙(审核中/待审核)、绿(已发布/启用/已通过)、默认(下架/已驳回)

### 交互规则

仅放置前面表格说不清的全局约束，不重复已有内容。必须包含：删除操作确认原则（所有删除动作均需L3确认弹窗，确认后生效）。

## 关键约束

- 不自创功能：所有功能描述严格来源于需求文档，新增闭环功能需标注并单独说明
- 不自加字段：表格字段仅来自功能描述原文，新增字段需与用户确认
- 不跳确认：每页输出后必须等用户确认。收到"可以""OK""继续"等明确信号后才进入下一页
- **命名对齐原文**：模块名、字段名、按钮名、状态名必须与需求文档原文或用户确认的术语一致，不可自行命名（如需求文档用"定义API"则不能用"新增API"）
- **交互复杂度控制**：优先简单交互方案（行列按钮、开关、下拉），拖拽、树形编辑等复杂交互需与用户确认后再用
- 模块名对齐：一级模块名等于需求文档模块名
- **后台菜单一致性**：当项目有≥2个后台管理页面时，所有页面必须使用完全一致的侧边栏菜单（菜单项列表相同、active状态正确、每个菜单项有跳转链接指向对应页面）。生成完所有后台页面后，逐页检查并统一
- **规格与原型同步**：每轮HTML原型改动经用户确认后，批量同步更新对应的规格MD文档。不必每改一个字段就更新一次——等用户说"这些可以了"或一轮修改结束时统一更新。规格中的字段配置表必须与实际HTML表单字段一一对应（数量、名称、类型、必填属性完全一致）
